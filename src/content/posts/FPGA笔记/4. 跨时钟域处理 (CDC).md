# 1. CDC 的三大核心挑战

在跨时钟域设计中，我们面临三个主要敌人：

1. **亚稳态 (Metastability)：信号在时钟沿变化，导致电平无法确定。
	- 对策： 同步器 (Synchronizer)。
2. **数据非一致性 (Data Incoherency)：** 针对**多比特**信号。由于总线中每一位的走线延迟不同（Skew），在第二个时钟域采样时，可能刚好有的位变了，有的位还没变，导致采集到一个“错误且合法”的中间值（比如 `00` -> `11` 变成了 `01`）。
	- 对策： 格雷码、FIFO、握手、MUX同步。
3. **信号丢失 (Signal Loss)：** 针对**快时钟域 -> 慢时钟域**。如果快时钟域产生一个单周期脉冲，而慢时钟域的采样周期太长，可能根本采不到这个脉冲。
	- 对策： 脉冲展宽、握手。

# 2. 单比特信号的处理策略 (Single-bit CDC)

单比特信号通常是控制信号（Flag, Enable, Pulse）。

## A. 慢时钟 -> 快时钟 (Slow to Fast)

这是最简单的情况。信号在快时钟看来维持得很久。
- **方法：** **电平同步器 (Level Synchronizer)**，即“打两拍”。
- **要求：** 输入信号的宽度必须大于快时钟周期的 1.5 倍（保守建议），确保快时钟至少能采到一次。
```txt
module level_sync (
    input  wire clk_dest,   // 目的时钟
    input  wire rst_n,      // 复位
    input  wire data_in,    // 来自源时钟域的异步信号
    output wire data_out    // 同步后的输出信号
);

    reg data_d1;
    reg data_d2;

    always @(posedge clk_dest or negedge rst_n) begin
        if (!rst_n) begin
            data_d1 <= 1'b0;
            data_d2 <= 1'b0;
        end else begin
            data_d1 <= data_in; // 第一拍：可能亚稳态
            data_d2 <= data_d1; // 第二拍：输出稳定数据
        end
    end

    assign data_out = data_d2;

endmodule
```
## B. 快时钟 -> 慢时钟 (Fast to Slow)

直接打拍会漏掉脉冲。必须处理。
- **方法 1：脉冲展宽 (Pulse Extension)**
	- 在快时钟域，一旦检测到脉冲，就拉高一个信号，直到收到慢时钟域的“反馈确认”才拉低（类似握手）。或者简单粗暴地计数器延时，保证高电平持续时间 > 慢时钟周期 * 1.5。
- **方法 2：脉冲同步器 (Pulse Synchronizer / Toggle Synchronizer)**
	- **原理：** 将快时钟域的“脉冲”转换为“电平翻转”（Toggle）。每来一个脉冲，信号就翻转一次。
	- **源时钟域：** 每来一个脉冲，寄存器状态翻转一次（0变1，或1变0）。
	- **跨时钟：** 将这个翻转后的信号（现已变为电平信号）传输到慢时钟域。
	- **目的时钟域：** 对该信号进行“打两拍”同步。
	- **边沿检测：** 在目的时钟域通过异或门（XOR）检测信号是否发生翻转，如果翻转则产生一个周期的脉冲。
```txt
module pulse_sync (
    input  wire clk_src,      // 源时钟（快）
    input  wire rst_n,
    input  wire pulse_in,     // 源域脉冲
    input  wire clk_dest,     // 目的时钟（慢）
    output wire pulse_out     // 目的域脉冲
);

    // --- 源时钟域：脉冲转翻转电平 ---
    reg toggle_reg;
    always @(posedge clk_src or negedge rst_n) begin
        if (!rst_n)
            toggle_reg <= 1'b0;
        else if (pulse_in)
            toggle_reg <= ~toggle_reg; // 检测到脉冲则翻转
    end

    // --- 跨时钟域：打两拍同步 ---
    reg sync_d1, sync_d2;
    always @(posedge clk_dest or negedge rst_n) begin
        if (!rst_n) begin
            sync_d1 <= 1'b0;
            sync_d2 <= 1'b0;
        end else begin
            sync_d1 <= toggle_reg;
            sync_d2 <= sync_d1;
        end
    end

    // --- 目的时钟域：边沿检测还原脉冲 ---
    reg sync_d3;
    always @(posedge clk_dest or negedge rst_n) begin
        if (!rst_n)
            sync_d3 <= 1'b0;
        else
            sync_d3 <= sync_d2; // 延迟一拍用于做边沿检测
    end

    // 只要sync_d2和sync_d3不同，说明源域有过脉冲
    assign pulse_out = sync_d2 ^ sync_d3; 

endmodule
```
## C. 注意事项
### 1. 禁止多比特信号使用“打两拍”
### 2. 源信号必须是寄存器输出

- 跨时钟域的信号，在离开源时钟域之前，最好是经过D触发器输出的（Clean Output）。
- **原因：** 如果源信号是组合逻辑输出（Glitchy），毛刺可能会被目的时钟误认为是有效信号并采样下来。
### 3. 输入脉冲间隔限制（针对脉冲同步）

- 在使用上述“翻转同步法”时，源时钟域的两个输入脉冲不能靠得太近。
- **限制：** 两个脉冲的间隔必须大于 `2 * T_dest_clk`（目的时钟周期）。否则，第一个翻转还没传输完，第二次翻转就来了，状态抵消，导致丢失。

# 3. 多比特信号的处理策略 (Multi-bit CDC)

## A. MUX 同步 / 采样限定(Data Qualification)

[5. 采样限定 Data Qualification](/posts/src\content\posts\FPGA笔记\5. 采样限定 Data Qualification/)

**MUX同步核心思想：** 数据本身**不**经过同步器，而是保持稳定。我们只同步一个单比特的“控制信号（Valid/Enable）”。只要 `Valid` 信号同步带来的延迟 > 数据总线的最大 Skew，数据就是稳定的。

**流程：**
1. **源时钟域：** 准备好多比特数据 `data_src`，保持数值不变。
2. **源时钟域：** 发送一个单比特信号 `data_valid`（高电平表示数据有效）。
3. **CDC：** 对 `data_valid` 进行“打两拍”同步，传到目的时钟域得到 `data_valid_dest`。
4. **目的时钟域：** 当检测到 `data_valid_dest` 为高电平时，可以直接采样 `data_src`。
**优点：** 简单，节省面积。 **缺点：** 传输效率低，必须保证数据在同步期间保持不动。
---

## B. 格雷码 (Gray Code)

**适用：**数值**连续变化**的计数器或指针（如 FIFO 指针）。

- **原理：** 任意两个相邻数值（如 5->6），格雷码只有 **1 bit** 发生跳变。
- **妙处：** 如果这 1 bit 发生亚稳态或采样错误，结果顶多是“停留在旧值”或“变为新值”，不会产生偏差巨大的“飞码”（如二进制 `011` -> `100` 可能错成 `111`）。
- **限制：** 只能用于计数累加场景，不能用于随机数据。

## D. 异步 FIFO (Asynchronous FIFO)

- **[7. 异步 FIFO (Asynchronous FIFO)](/posts/src\content\posts\FPGA笔记\7. 异步 FIFO (Asynchronous FIFO)/)**
**适用：**
- 高速、大数据量、连续传输。
- 大量数据传输、高带宽、源/目时钟频率差异大、数据突发（Burst）。这是**最通用、最标准**的解法。

**核心结构：**
1. **双口 RAM (Dual-port RAM)：** 用于存储数据。
2. **写指针 (Write Pointer)：** 在源时钟域，转换成格雷码后传给读侧。
3. **读指针 (Read Pointer)：** 在目的时钟域，转换成格雷码后传给写侧。
4. **判空/判满逻辑：** 比较两个时钟域传过来的格雷码指针。

**关键点：** 将多比特的数据传输问题，转化为了单比特（满/空标志位）的生成问题。

# 4. CDC 重新收敛 (CDC Reconvergence)

这是一个极易被忽视的严重错误。

- **场景：** 信号 A 从时钟域 1 发出，分成了两路（或者控制了两个不同的信号 B 和 C）。B 和 C 分别通过了各自的同步器进入时钟域 2。在时钟域 2 内部，逻辑又把 B 和 C 组合起来使用（如 `D = B & C`）。
- **问题：** 由于两个同步器的延迟是不确定的（可能是 2 拍，也可能是 3 拍，取决于亚稳态恢复时间），B 和 C 到达时钟域 2 的时间差是不确定的。
- **后果：** 它们重新组合（Reconvergence）时，可能会产生功能错误或意外的毛刺。
- **对策：** 避免分头同步！尽量在源时钟域把 B 和 C 组合好，变成一个信号，然后再跨时钟域；或者使用格雷码/FIFO 整体传输。

# 5. 约束与验证 (Constraints & Verification)

工程上，仅仅写好 Verilog 是不够的，必须告诉工具怎么处理。

## A. 时序约束 (Constraints)

[3. 时序约束和物理约束](/posts/src\content\posts\FPGA笔记\3. 时序约束和物理约束/)
在 Vivado (`.xdc`) 或 SDC 中，你需要明确告诉工具哪些路径不需要做同步时序分析：

- `set_clock_groups -asynchronous -group {clk_A} -group {clk_B}`: 彻底切断两个时钟域的时序关系（慎用，除非全都是 FIFO/同步器隔离）。
- `set_false_path -from [get_cells reg_a] -to [get_cells reg_b]`: 针对特定的打拍路径，忽略时序检查。
- `set_max_delay -datapath_only`: **针对格雷码或 MUX 同步的数据总线**，虽然不需要检查 setup/hold，但需要限制最大线延，防止Skew 太大。
## B. CDC 静态检查 (CDC Linting)

常规的仿真（Simulation）很难发现 CDC 问题，因为仿真器通常不模拟亚稳态。
**静态时序分析（STA）**：CDC路径无法用STA验证，必须通过**同步器设计**和**功能仿真**来保证正确性。STA工具仅会检查同步器内部的时序（如两个触发器之间的路径）。

必须使用专用工具（如 SpyGlass CDC, Questa CDC, 或 Vivado 自带的 Report CDC）：
- 它会检查是否所有跨域路径都加了同步器。
- 它会检查是否存在“重新收敛”结构。
- 它会检查复位与时钟的搭配问题。

# 6.注意

## Q1: 为什么多比特信号不能直接打两拍？

- **答案关键词：** 路径延迟不同（Skew）、数据不一致（Incoherence）、亚稳态导致各位恢复时间不同。
- **解释：** 比如从 `00` 变 `11`，可能被误采为 `01` 或 `10`，导致功能错误。
## Q2: 异步 FIFO 的深度怎么计算？（计算题）

- **场景：** 写时钟 100MHz，读时钟 80MHz，突发写入 100 个数据（Burst Length），每两个突发之间有空闲。问 FIFO 最小深度？
- **思路：** 考虑最坏情况（背靠背突发）。计算在“写完这 100 个数据的时间内”，读侧能读走多少个数据。剩下的就是必须存储在 FIFO 里的量。
- **公式概念：** $Depth = Burst\_Length - (Burst\_Length \times \frac{f_{read}}{f_{write}})$ （需根据具体情况推导）。
## Q3: 格雷码同步有什么限制？

- **答案：** 只能用于**单调递增/递减**的计数器信号。如果数据是随机跳变的（如 ADC 采样值），不能用格雷码，必须用 FIFO 或握手。
## Q4: 异步 FIFO 中，判空和判满的指针比较逻辑是什么？

- **判空 (Empty)：** 读指针 == 写指针（在读时钟域比较）。
- **判满 (Full)：** 写指针 == 读指针（在写时钟域比较）。
    - _进阶追问：_ 格雷码判满时，高两位的位特征是什么？
    - _答案：_ 最高位不同，次高位不同，其余位相同（针对常见的格雷码算法）。
## Q5: 什么是虚空（False Empty）和虚满（False Full），它们有害吗？

- **答案：**
    - **虚空：** 写指针同步到读域需要时间，导致读域看到的写指针是“旧”的。明明有新数据写入了，但读侧还以为是空的。**结果：** 只是降低一点吞吐率，功能**安全**。
    - **虚满：** 同理，读指针同步回写域是“旧”的。明明读走了数据，写侧还以为是满的。**结果：** 只是让写侧多等一会儿，功能**安全**。
    - **结论：** 异步 FIFO 的设计是“悲观”的，永远偏向安全。
