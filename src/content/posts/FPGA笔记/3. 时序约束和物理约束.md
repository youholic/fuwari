# 一. 时序约束

**定义：**
- **时序约束是设计者强加给电路的时序目标和限制条件。**
**作用：**
- 指导布局布线；
- 驱动逻辑优化（约束决定了电路的最终结构）；
- 时序分析与验收（计算 Slack（裕量）的基准）
## 1. 时钟约束 (Clock Constraints) 

这是最重要、最基础的约束。没有时钟约束，所有的 Setup/Hold 分析都无从谈起。

- **基本时钟 (Primary Clock)**：定义输入引脚上的时钟。
```txt
create_clock -name clk -period 10 [get_ports clk_in]  # 10ns周期（100MHz）
```
- **生成时钟 (Generated Clock)**：如果芯片内部有 PLL/MMCM 或者分频逻辑，你需要定义生成时钟。
```txt
create_generated_clock -name clk_div2 -source [get_pins pll/CLKOUT] -divide_by 2 [get_pins pll/CLKOUT_DIV2]
```
- **时钟不确定性 (Clock Uncertainty / Jitter)**：告诉工具：“时钟并不完美，会有抖动（Jitter），请预留 0.1ns 的余量。”这会让工具在计算 Slack 时更保守（Setup 变难）。
```txt
set_clock_uncertainty 0.5 [get_clocks clk]  # 设置500ps时钟不确定性
```
- **跨时钟域约束**
```txt
set_clock_groups -asynchronous -group {clk1} -group {clk2}  # 声明clk1和clk2异步
```

## 2. I/O 约束 (Input/Output Delay) 

很多人误以为 set_input_delay 是设置“FPGA 内部输入管脚的延时”。大错特错！
**核心概念：约束描述的是FPGA 外部的世界。**

- **Input Delay**：告诉 FPGA，“外部芯片和 PCB 走线已经消耗了多少时间，留给 FPGA 内部 Setup 的时间只剩多少了”。
- **Output Delay**：告诉 FPGA，“外部接收芯片要求建立时间是多少，PCB 走线要跑多久，所以 FPGA 必须在什么时间点之前把数据吐出来”。

### **1. 如何约束输入延迟（Input Delay）？**

输入延迟约束定义**输入信号相对于时钟边沿的到达时间**，确保外部信号满足FPGA内部触发器的时序要求。
#### **输入延迟约束语法**
```txt
set_input_delay -clock [get_clocks clk] -max 2.5 [get_ports data_in]
set_input_delay -clock [get_clocks clk] -min 1.0 [get_ports data_in]
```
- `-max`：信号最晚到达时间（相对于时钟边沿），用于检查Setup时间。
- `-min`：信号最早到达时间，用于检查Hold时间。
### **2. 如何约束输出延迟（Output Delay）？**

输出延迟约束定义**FPGA输出信号相对于时钟边沿的稳定时间**，确保外部器件能正确采样。
#### **输出延迟约束语法**
```txt
set_output_delay -clock [get_clocks clk] -max 4.0 [get_ports data_out]
set_output_delay -clock [get_clocks clk] -min 0.5 [get_ports data_out]```
```
- `-max`：外部器件要求信号最晚稳定时间（用于Setup检查）。
- `-min`：外部器件要求信号最早稳定时间（用于Hold检查）。
## 3. 时序例外 (Timing Exceptions) —— 也就是刚才讲的

- **False Path**：`set_false_path`（异步复位、跨时钟域）。
- **Multicycle Path**：`set_multicycle_path`（多周期运算）。
- **Max/Min Delay**：直接强行规定两点之间的最大/最小延时（用的较少，通常用于纯组合逻辑路径）。
## 4. 注意

a. **如果不加任何约束，综合出来的电路能用吗？**
- **答：** 功能逻辑是对的，但速度完全不可控。工具会“躺平”，随便布局布线，可能导致本来能跑 100MHz 的电路最后只能跑 10MHz。而且没有 IO 约束，接口时序无法保证。
b. **什么是过约束 (Over-constraining)？有什么优缺点？**
- **答：** 比如目标跑 100MHz (10ns)，我故意约束成 120MHz (8.33ns)。
- **优点：** 逼迫工具更努力地优化布局布线，如果能满足 120MHz，那在 100MHz 下裕量（Slack）非常充足，系统更稳健。
- **缺点：** 编译时间变长；如果约束太离谱，工具可能会为了满足 Setup 而疯狂牺牲面积或 Hold 时间，导致结果反而变差。
c. **为什么 Input Delay 约束的是外部延时，而不是内部？**
- **答：** 因为内部延时是工具自己计算和控制的（它知道内部走线有多长）。工具不知道的是**外部环境**。只有你告诉它外部环境占用了多少预算，它才知道内部还剩多少预算可用。

# 二. 物理约束 (Physical Constraints)

- **管脚约束 (Pin Assignment)**：哪个信号连哪个物理引脚（Pin Loc）。
- **电平标准 (IO Standard)**：LVCMOS33, LVDS, SSTL 等。
- **区域约束 (Floorplanning / Pblock)**：强制把某个模块的逻辑放在芯片的左上角。这通常用于解决拥塞（Congestion）或极端的时序问题。

# 三. 时序例外

## 1. 虚假路径 (False Path)

- **定义：** 电路中物理连接存在，但在实际功能运行中，信号根本不会通过这条路径传播，或者即使传播了，我们也不关心它的时序（结果会被丢弃）。
- **三大场景**：
1. **跨时钟域 (CDC, Clock Domain Crossing)：** 从时钟域 A 到时钟域 B 的直接路径。因为两个时钟没有相位关系，做静态时序分析没有意义。我们通常会在中间加同步器（打两拍），然后对这段路径设置 `set_false_path`
2. **静态配置寄存器 (Static Configuration Register)：** 芯片上电初始化后就不再变化的寄存器（比如模式选择位）。这些信号可能会跑很远，延时很大，但因为它们是“常量”，永远不会变，所以不需要满足 Setup/Hold。
3. **逻辑上不可能的路径 (Logically Exclusive)：** 例如两个多路复用器（MUX）由同一个 Select 信号控制，使得某些数据通路在逻辑上永远无法打通。

**Tcl 约束命令 (SDC/XDC)：**
```tcl
set_false_path -from [get_clocks clk_a] -to [get_clocks clk_b]
```
---

## 2. 多周期路径 (Multi-cycle Path)

- **定义：** 默认情况下，工具认为所有数据都必须在1个时钟周期内跑完。但如果逻辑太复杂（比如大位宽乘法），或者设计意图本身就是慢速传输（每3个周期才采一次样），我们就可以告诉工具：“这条路允许跑 N 个周期”。
- **注意：**设置了 Setup 多周期后，Hold 也需要调整，通常 Setup 设置为 $N$，Hold 就要设置为 $N-1$。

**Tcl 约束命令 (SDC/XDC)：**
```tcl
# 允许数据跑 2 个周期 (Setup 检查推迟到第 2 个沿)
set_multicycle_path -setup 2 -from [get_pins A/C] -to [get_pins B/D]

# 关键一步！把 Hold 检查拉回第 0 个沿 (前移 1 个周期)
set_multicycle_path -hold 1  -from [get_pins A/C] -to [get_pins B/D]
```
---

## 3. 什么时候用多周期路径？（工程实例）

假设你的系统跑 100MHz (10ns)，但你需要做一个复杂的 32位乘法，组合逻辑延时高达 18ns。

- **方法A (流水线)**：中间插寄存器。优点是吞吐率高（每周期都能出结果），缺点是占面积。

- **方法B (多周期)**：如果你的系统逻辑允许每 2 个周期才算一次结果（吞吐率减半），那就不需要改代码插寄存器，直接写一条约束 `set_multicycle_path 2`。这省面积，但牺牲了性能。
