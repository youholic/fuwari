握手协议（Handshake Protocol）是 CDC 处理中最可靠、最稳健，但也是**效率最低（延迟最大）**的方法。

它是一种**闭环（Closed-loop）传输方式。与之前的“打两拍”或“FIFO”（开环，发了就不管了）不同，握手协议要求接收端必须给发送端一个“回执（ACK）”，发送端收到回执后才算完成一次传输。

---

# 一、核心思想与流程

最经典的握手是**四相握手（4-Phase Handshake）**，也叫全握手（Full Handshake）。
## 1. 角色定义

- **Source (发送端):** 发送数据 `Data` 和请求信号 `Req`。
- **Destination (接收端):** 接收数据，并返回应答信号 `Ack`。
## 2. 详细步骤 (The 4 Phases)

1. Phase 1 (Assert Req):
    源端准备好数据 Data，将 Req 拉高（置1）。此时数据必须保持稳定。
2. Phase 2 (Assert Ack):
    目的端通过同步器检测到 Req 变为高电平后，采样数据，并将 Ack 拉高（置1）。
3. Phase 3 (De-assert Req):
    源端通过同步器检测到 Ack 变为高电平后，知道数据已安全到达，将 Req 拉低（置0）。
4. Phase 4 (De-assert Ack):
    目的端通过同步器检测到 Req 变低后，将 Ack 拉低（置0）。也就是回到初始状态，准备下一次传输。

# 二、 Verilog 代码实现

为了清晰展示，我将发送端（Master）和接收端（Slave）写在同一个模块中，但逻辑上是严格隔离的。使用了状态机（FSM）来实现，这是最标准的写法。

```txt
module cdc_handshake (
    // --- 源时钟域 (Source Domain) ---
    input  wire clk_src,
    input  wire rst_n_src,
    input  wire [7:0] data_in,
    input  wire valid_in,       // 上游发来的发送指令
    output reg  busy,           // 告诉上游：我正在握手，别发新数据

    // --- 目的时钟域 (Dest Domain) ---
    input  wire clk_dest,
    input  wire rst_n_dest,
    output reg  [7:0] data_out,
    output reg  valid_out       // 告诉下游：新数据准备好了
);

    // ============================================================
    // 信号定义
    // ============================================================
    
    // 跨时钟域信号
    reg  req_src;       // 源发出的请求
    wire req_dest_sync; // 目的端看到的同步后的请求
    
    reg  ack_dest;      // 目的发出的应答
    wire ack_src_sync;  // 源端看到的同步后的应答

    reg [7:0] data_hold_reg; // 源端数据保持寄存器

    // ============================================================
    // 1. 同步器模块 (Double Flop Synchronizers)
    // ============================================================
    
    // 将 req 从 src 同步到 dest
    reg req_d1, req_d2;
    always @(posedge clk_dest or negedge rst_n_dest) begin
        if (!rst_n_dest) {req_d2, req_d1} <= 2'b0;
        else             {req_d2, req_d1} <= {req_d1, req_src};
    end
    assign req_dest_sync = req_d2;

    // 将 ack 从 dest 同步到 src
    reg ack_d1, ack_d2;
    always @(posedge clk_src or negedge rst_n_src) begin
        if (!rst_n_src) {ack_d2, ack_d1} <= 2'b0;
        else            {ack_d2, ack_d1} <= {ack_d1, ack_dest};
    end
    assign ack_src_sync = ack_d2;

    // ============================================================
    // 2. 源端状态机 (Source FSM)
    // ============================================================
    localparam S_IDLE     = 2'd0;
    localparam S_REQ_HIGH = 2'd1;
    localparam S_REQ_LOW  = 2'd2;

    reg [1:0] src_state;

    always @(posedge clk_src or negedge rst_n_src) begin
        if (!rst_n_src) begin
            src_state     <= S_IDLE;
            req_src       <= 1'b0;
            busy          <= 1'b0;
            data_hold_reg <= 8'd0;
        end else begin
            case (src_state)
                S_IDLE: begin
                    if (valid_in) begin
                        data_hold_reg <= data_in; // 锁存数据
                        req_src       <= 1'b1;    // Phase 1: 拉高 Req
                        busy          <= 1'b1;    // 开始忙碌
                        src_state     <= S_REQ_HIGH;
                    end else begin
                        busy <= 1'b0;
                    end
                end

                S_REQ_HIGH: begin
                    // 等待 Ack 回来 (Phase 3 的前奏)
                    if (ack_src_sync) begin
                        req_src   <= 1'b0;        // Phase 3: 拉低 Req
                        src_state <= S_REQ_LOW;
                    end
                end

                S_REQ_LOW: begin
                    // 等待 Ack 变低 (Phase 4 完成) - 确保彻底完成握手
                    if (!ack_src_sync) begin
                        busy      <= 1'b0;        // 解除忙碌
                        src_state <= S_IDLE;
                    end
                end
            endcase
        end
    end

    // ============================================================
    // 3. 目的端状态机 (Destination FSM)
    // ============================================================
    localparam D_IDLE     = 2'd0;
    localparam D_ACK_HIGH = 2'd1;
    
    reg [1:0] dest_state;

    always @(posedge clk_dest or negedge rst_n_dest) begin
        if (!rst_n_dest) begin
            dest_state <= D_IDLE;
            ack_dest   <= 1'b0;
            data_out   <= 8'd0;
            valid_out  <= 1'b0;
        end else begin
            case (dest_state)
                D_IDLE: begin
                    valid_out <= 1'b0; // 默认无效
                    // 检测到 Req 变高 (Phase 2 的前奏)
                    if (req_dest_sync) begin
                        data_out   <= data_hold_reg; // 采样数据（此时数据绝对稳定）
                        valid_out  <= 1'b1;          // 通知下游
                        ack_dest   <= 1'b1;          // Phase 2: 拉高 Ack
                        dest_state <= D_ACK_HIGH;
                    end
                end

                D_ACK_HIGH: begin
                    valid_out <= 1'b0; // 脉冲只维持一个周期
                    // 等待 Req 变低 (Phase 4 的前奏)
                    if (!req_dest_sync) begin
                        ack_dest   <= 1'b0;          // Phase 4: 拉低 Ack
                        dest_state <= D_IDLE;
                    end
                end
            endcase
        end
    end

endmodule
```

# 三、 优缺点与适用场景

## 1. 优点

- **绝对安全：** 数据不依赖于时序参数（Max Delay），而是依赖于逻辑反馈。
- **自适应频率：** 无论是快传慢，还是慢传快，或者两个时钟频率动态变化，它都能完美工作。
- **自带背压（Backpressure）：** 如果接收端很忙（不拉高 ACK），发送端会自动等待（Busy 信号一直为高），不会发生数据覆盖或丢失。
## 2. 缺点

- **延迟极大：** 完成一次传输至少需要 3次跨时钟同步（Req过去，Ack回来，Req拉低再检测）。假设同步器需要2拍，那么一次握手至少消耗 **2 + 2 + 2 = 6 个周期**（以较慢的时钟为准）。
- **吞吐率低：** 无法连续传输数据。
## 3. 适用场景

- 低频控制信号的参数配置（寄存器配置）。
- 两个完全异步且不了解对方频率的模块之间的命令交互。
- 对数据正确性要求极高，但对速度不敏感的场合。

# 四、 进阶：面试常考点

## 1. "两相握手" (2-Phase Handshake) 是什么？

Verilog 代码中展示的是四相握手（Req 高 -> Ack 高 -> Req 低 -> Ack 低）。
两相握手是指：
- Req 翻转（0变1 或 1变0）代表一次请求。
- Ack 翻转（0变1 或 1变0）代表一次应答。
- **优点：** 速度快一倍，不需要等待“归零”。
- **缺点：** 逻辑设计稍微复杂一点（需要做双边沿检测或者异或逻辑），且在某些复位场景下容易状态不同步。
## 2. 握手协议需要 Data Path 约束吗？

虽然理论上握手协议通过逻辑等待保证了稳定，但在工程上，依然建议对 `data_hold_reg` 到 `data_out` 的路径设置 `set_max_delay -datapath_only`。
- **原因：** 防止 EDA 工具把数据线布得绕地球一圈。虽然逻辑上会等 Ack，但如果数据线延迟长到超过了 Req 的同步时间（极罕见），理论上还是有风险。约束一个宽松的值（如 2-3 个时钟周期）即可。
## 3. 为什么代码里采样 `data_hold_reg` 时没有打两拍？

代码中：`data_out <= data_hold_reg;`
- 这是**正确**的。因为当 `req_dest_sync` 在目的域变为 1 时，源域的 `req_src` 已经经过了至少 2 个目的时钟周期的传播。而 `data_hold_reg` 是在 `req_src` 拉高之前就已经锁存稳定的。
- 只要数据路径延迟小于同步器的延迟（通常肯定满足），此时 `data_hold_reg` 在目的端看来就是纯直流电平，直接采，绝对安全。
