# 一、 复位信号

### 1. 同步复位 (Synchronous Reset)

同步复位意味着复位信号只有在时钟沿（通常是上升沿）到来时才会被采样和生效。

#### **代码示例**
```txt
always @(posedge clk) begin
    if (rst) begin
        q <= 1'b0;
    end else begin
        q <= d;
    end
end
```
#### 硬件实现机制

在大多数FPGA架构（如Xilinx 7系列或UltraScale）中，如果使用同步复位，综合工具通常会将复位信号作为输入逻辑的一部分，通过**LUT（查找表）**来实现复位功能，或者利用触发器（Flip-Flop）自带的同步复位/置位端口（如果该架构支持）。
#### 优点

- **抗干扰能力强：** 复位信号上的毛刺（Glitches）如果未覆盖时钟边沿，会被时钟自然过滤掉，不会导致系统误复位。
- **利于时序分析：** 复位信号被视为普通的“数据路径”信号，标准的静态时序分析（STA）工具可以轻松处理。
- **利于重定时（Retiming）：** 现代FPGA工具（如Vivado）在优化时序时，可以更容易地在逻辑路径中移动寄存器位置。如果复位是同步的，工具可以自由移动寄存器；如果是异步的，寄存器位置通常被固定。
#### 缺点

- **脉冲宽度要求：** 复位信号的持续时间必须大于一个时钟周期，否则可能在时钟沿之间被漏掉。
- **依赖时钟：** 如果时钟停转（例如在低功耗模式下），系统无法进入复位状态。
- **资源消耗（视架构而定）：** 在某些老旧或特定的FPGA架构中，如果触发器没有专用的同步复位端口，综合工具必须在数据输入端增加组合逻辑（LUT资源）来实现复位，这可能增加逻辑层级，影响时序。
---

### 2. 异步复位 (Asynchronous Reset)

异步复位意味着无论时钟处于什么状态，只要复位信号有效，寄存器立即被复位。
#### 代码示例
```txt
always @(posedge clk or posedge rst) begin
    if (rst) begin
        q <= 1'b0;
    end else begin
        q <= d;
    end
end
```
#### 硬件实现机制

异步复位直接连接到触发器的专用 **CLR（Clear）或PRE（Preset）端口**。 这通常是全局路由资源，不经过LUT。
#### 优点

- **响应速度快：** 不依赖时钟，立即生效。
- **独立性：** 即使时钟丢失或尚未稳定，电路也能被强制复位到已知状态。
- **节省逻辑资源：** 直接利用触发器的专用引脚，不占用数据路径的LUT资源（假设架构支持）。
#### 缺点

- **易受毛刺影响：** 复位线上的任何噪声或毛刺都可能导致系统瞬间误复位，这在工业环境或高辐射环境中是致命的。
- **释放（De-assertion）时的亚稳态风险：** 这是最核心的问题。虽然进入复位是异步的，但**退出复位**必须考虑时序。如果复位信号在时钟的建立时间（Setup time）和保持时间（Hold time）窗口内释放，触发器的输出将进入亚稳态（Metastability），导致系统状态不可预测。

这个时序窗口被称为**恢复时间（Recovery Time）和撤销时间（Removal Time）**，类比于数据的Setup和Hold时间。

### 3. 最佳实践：异步复位，同步释放 (Asynchronous Assert, Synchronous De-assert)

为了结合上述两者的优点并消除缺点，工业界（尤其是ASIC和高可靠性FPGA设计）通常采用“异步复位，同步释放”的策略。

这种方案使用一个被称为**复位桥（Reset Bridge）或复位同步器**的电路。
#### 核心思想

1. **异步生效：** 当外部复位信号到来时，系统立即复位（不依赖时钟）。
2. **同步释放：** 当外部复位信号撤销时，复位信号在FPGA内部被时钟同步化，确保在时钟边沿的安全区域释放，避免亚稳态。
#### 硬件电路与代码

```txt
// 异步复位，同步释放 模块示例
module reset_synchronizer (
    input  wire clk,
    input  wire rst_n_async, // 外部异步复位（低电平有效）
    output reg  rst_n_sync   // 内部同步化后的复位
);

    reg rst_n_meta;

    always @(posedge clk or negedge rst_n_async) begin
        if (!rst_n_async) begin
            rst_n_meta <= 1'b0; // 异步复位：立即拉低
            rst_n_sync <= 1'b0;
        end else begin
            rst_n_meta <= 1'b1; // 同步释放：打两拍
            rst_n_sync <= rst_n_meta;
        end
    end

endmodule
```

**为什么这样有效？**

- 当 `rst_n_async` 拉低时，两个触发器通过专用CLR端口立即清零，`rst_n_sync` 立即变低。系统迅速复位。
- 当 `rst_n_async` 变高（释放）时，第一个触发器 `rst_n_meta` 的D端变为高电平。在下一个时钟沿，如果发生亚稳态，它只发生在 `rst_n_meta` 上。
- 第二个触发器 `rst_n_sync` 会在再下一个时钟沿采样 `rst_n_meta` 的稳定状态。此时，复位信号对于后续电路来说，是一个完全与时钟同步的信号，不存在Recovery/Removal违例。

### 4.复位信号的约束
#### 1. 同步复位的约束：视为普通数据信号

对于完全的同步复位（`always @(posedge clk) if (rst) ...`），复位信号被综合工具视为普通的**数据路径（Data Path）**。

- **处理方式：** 不需要特殊的异常约束。时序分析工具会自动计算从寄存器的 `Q` 端（或输入引脚）到目标寄存器 `D` 端的路径延迟，并按照建立时间（Setup）和保持时间（Hold）来检查。
- **注意点：** 如果同步复位的扇出（Fan-out）极大（例如复位了几万个触发器），会导致时序违例。此时需要通过 `MAX_FANOUT` 属性进行物理优化，或者使用时钟使能（CE）树来分担负载。
- 
#### 2. 异步复位的约束：恢复与撤销时间

异步复位最危险的地方在于其**释放（De-assertion）**瞬间。如果释放时刻紧贴着时钟沿，触发器可能进入亚稳态。

- **核心约束指标：**
    - **恢复时间 (Recovery Time):** 类似于建立时间，指复位释放必须在时钟有效沿到来之前的一段时间内完成。
    - **撤销时间 (Removal Time):** 类似于保持时间，指复位释放必须在时钟有效沿到来之后保持一段时间。
- **约束方法：** 大部分现代EDA工具（如Vivado）默认会对异步复位端口进行 Recovery 和 Removal 检查。你只需要确保复位路径被正确定义。
#### 3. “异步复位，同步释放”的特殊处理

这是目前最主流的方案，但在约束时需要格外小心，避免工具产生错误的警告或忽略关键路径。

##### A. 复位同步器（Reset Bridge）的异步路径

在异步复位同步器的第一个触发器上，外部输入的 `rst_n_async` 是异步的，必然会产生 Recovery 违例。

- **处理方式：** 应当对该路径设置 **False Path**（伪路径），告诉工具忽略这个不可控的异步输入点。
- **XDC 示例：**
	```txt
    # 假设异步复位输入引脚名为 sys_rst_n
    set_false_path -from [get_ports sys_rst_n] -to [get_pins -hier *reset_sync_reg*/PRE]
    ```
##### B. 同步化后的复位信号

一旦复位信号通过了同步器（打过两拍），它就变成了同步信号。

- **处理方式：** 此时**不应**再设为 False Path。工具必须分析从同步器输出到全系统触发器的时序，以确保所有寄存器在同一个时钟周期内同步退出复位。

#### 4. 扇出（Fan-out）与全局时钟网络

复位信号通常要驱动芯片内几乎所有的寄存器，这会带来巨大的扇出压力。

- **利用全局时钟资源 (BUFG):** 在 Xilinx FPGA 中，建议将同步后的复位信号驱动到一个 `BUFG` 上。全局时钟网络具有极低的偏斜（Skew），能确保复位信号几乎同时到达芯片的各个角落。
    ```txt
    // 在代码中强制调用BUFG（可选）
    BUFG rst_bufg_inst (.I(rst_n_sync_internal), .O(rst_n_sync_global));
    ```
- **控制集 (Control Sets) 优化:** 如果一个 Slice 里的触发器混合了不同的复位信号，会导致资源浪费。约束中可以通过设置控制集阈值来优化布局。
#### 5. 跨时钟域复位 (Asynchronous Reset across CDC)

如果你的设计中有多个时钟域（例如 `clk_100M` 和 `clk_200M`）：

- **切忌：** 用同一个同步后的复位信号去驱动所有时钟域。
- **正确做法：** 必须为**每个时钟域独立设计**一个“异步复位，同步释放”的同步器。每个同步器都要针对其对应的时钟进行约束。

#### 6. 总结 Checklist

| **复位类型**   | **约束操作**                 | **关键命令/属性**                                      |
| ---------- | ------------------------ | ------------------------------------------------ |
| **纯同步复位**  | 无需特殊处理，按数据路径分析           | `create_clock` (基础时钟定义)                          |
| **外部异步输入** | 设为 False Path，防止误报       | `set_false_path -from [get_ports rst_pin]`       |
| **同步释放路径** | 必须进行 Recovery/Removal 检查 | 默认开启，需确保有时钟定义                                    |
| **高扇出路径**  | 限制最大扇出或使用全局时钟            | `set_property MAX_FANOUT 500 [get_nets rst_net]` |

### 5. Xilinx 与 Intel (Altera) 的架构差异建议

虽然“异步复位，同步释放”是通用准则，但不同的FPGA厂商有不同的推荐偏好，这取决于其底层硬件结构。

- **Xilinx (AMD):** 强烈推荐尽量使用**同步复位**。
    - Xilinx的DSP Slice和Block RAM等硬核通常只支持同步复位（或同步复位性能更好）。
    - 使用同步复位允许综合工具将复位逻辑吸收到LUT中，或者利用控制集（Control Sets）优化，不仅减少布线拥塞，还允许更高级的物理优化（如Retiming）。
    - 官方文档（如UG901）明确指出：如果不要求必须异步复位，请使用同步复位。
- **Intel (Altera):** 传统上更倾向于**异步复位**。
    - 其LAB（Logic Array Block）结构通常对异步信号有很好的支持。
    - 但在新的HyperFlex架构（如Stratix 10, Agilex）中，为了利用Hyper-Retiming技术达到超高性能，**同步复位**也变得至关重要。
### 总结

| **特性**   | **同步复位**        | **异步复位**             | **异步复位，同步释放 (推荐)** |
| -------- | --------------- | -------------------- | ------------------ |
| **复位生效** | 需等待时钟沿          | 立即生效                 | 立即生效               |
| **时钟依赖** | 高               | 无                    | 仅在释放时依赖            |
| **抗毛刺**  | 好 (时钟滤波)        | 差 (毛刺即复位)            | 差 (需外部滤波)          |
| **时序风险** | 低 (标准数据路径)      | 高 (Recovery/Removal) | 低 (内部已同步)          |
| **主要用途** | 数据流处理、DSP算法、流水线 | 全局系统复位、紧急停机          | 系统主复位、大多数通用逻辑      |
