## 文件列表及说明

1. **Makefile** (314B, 11行)
   - Linux 内核模块构建文件
   - 定义编译目标，包含多个子模块（sdr.o、openofdm_rx、openofdm_tx、tx_intf、rx_intf、xpu）

2. **make_all.sh** (3.3KB, 135行)
   - 构建脚本
   - 设置 Xilinx 环境，编译各子模块，生成内核模块

3. **hw_def.h** (18KB, 465行)
   - 硬件定义头文件
   - 包含硬件类型枚举、寄存器地址定义、接口模式定义、驱动 API 结构体等

4. **sdr.h** (18KB, 533行)
   - SDR（软件定义无线电）头文件
   - 定义数据结构（如 `openwifi_priv`、`openwifi_ring`、`openwifi_vif`）、LED 控制、RF 操作接口等

5. **sdr.c** (115KB, 2761行)
   - SDR 驱动主实现文件
   - 实现 mac80211 接口、DMA 传输、RF 控制、中断处理等核心功能

6. **sysfs_intf.c** (46KB, 1270行)
   - sysfs 接口实现
   - 提供用户空间通过 sysfs 访问驱动的接口（如 IQ 数据写入、寄存器读写等）

7. **sdrctl_intf.c** (20KB, 450行)
   - sdrctl 测试模式接口实现
   - 处理 nl80211 测试模式命令（如设置/获取 gap、slice_idx、寄存器读写等）

这些文件共同构成 OpenWifi 的 Linux 内核驱动，用于在 Xilinx FPGA 平台上实现 WiFi 功能。

### 2. 依赖关系详解

#### hw_def.h → 所有其他文件
- 被 `sdr.c`、`sysfs_intf.c`、`sdrctl_intf.c` 包含
- 定义：
  - `tx_intf_driver_api`、`rx_intf_driver_api`
  - `openofdm_tx_driver_api`、`openofdm_rx_driver_api`
  - `xpu_driver_api`
  - 寄存器地址宏定义

#### sdr.h → sdr.c, sysfs_intf.c, sdrctl_intf.c
- 定义核心数据结构：
  - `struct openwifi_priv`（驱动私有数据）
  - `struct openwifi_ring`（DMA 缓冲区管理）
  - `struct openwifi_vif`（虚拟接口）

#### sdr.c → 子模块驱动
- 通过 extern 声明使用子模块 API：
```bash:87:openwifi-master/driver/sdr.c
extern struct tx_intf_driver_api *tx_intf_api;
extern struct rx_intf_driver_api *rx_intf_api;
extern struct openofdm_tx_driver_api *openofdm_tx_api;
extern struct openofdm_rx_driver_api *openofdm_rx_api;
extern struct xpu_driver_api *xpu_api;
```
- 在 `openwifi_start()` 中初始化各子模块：
```bash:1621:openwifi-master/driver/sdr.c
rx_intf_api->hw_init(priv->rx_intf_cfg,8,8);
tx_intf_api->hw_init(priv->tx_intf_cfg,8,8,priv->fpga_type);
openofdm_tx_api->hw_init(priv->openofdm_tx_cfg);
openofdm_rx_api->hw_init(priv->openofdm_rx_cfg);
xpu_api->hw_init(priv->xpu_cfg);
```

#### sysfs_intf.c → sdr.c
- 通过 `platform_get_drvdata()` 获取 `openwifi_priv`
- 提供 sysfs 属性，供用户空间读写驱动状态和配置
- 使用 `struct openwifi_priv *priv` 访问驱动数据

#### sdrctl_intf.c → sdr.c, hw_def.h
- 实现 `openwifi_testmode_cmd()` 处理 nl80211 测试模式命令
- 通过 `xpu_api`、`tx_intf_api`、`rx_intf_api` 等访问硬件
- 使用 `hw_def.h` 中的命令定义

### 3. 编译依赖关系

#### Makefile
```bash:3:openwifi-master/driver/Makefile
obj-m += sdr.o openofdm_rx/openofdm_rx.o openofdm_tx/openofdm_tx.o tx_intf/tx_intf.o rx_intf/rx_intf.o xpu/xpu.o
```
- 定义编译目标：主模块 `sdr.o` 和 5 个子模块
- `sdr.c` 编译为 `sdr.o`，包含 `sysfs_intf.c` 和 `sdrctl_intf.c` 的功能（通过链接）

#### make_all.sh
- 按顺序编译各子模块，最后编译主模块
- 生成 `pre_def.h` 和 `git_rev.h`（被 `sdr.c` 包含）

### 4. 数据流向

```
用户空间应用
    ↓ (sysfs / nl80211)
sysfs_intf.c / sdrctl_intf.c
    ↓ (访问 priv 结构)
sdr.c (主驱动)
    ↓ (调用 API)
子模块驱动 (tx_intf, rx_intf, openofdm_tx, openofdm_rx, xpu)
    ↓ (寄存器操作)
FPGA 硬件
```

### 5. 关键接口点

1. API 接口：`hw_def.h` 定义函数指针结构体，子模块实现并导出
2. 数据结构：`sdr.h` 定义共享结构体，各文件通过 `openwifi_priv` 访问
3. 编译链接：`Makefile` 将各模块编译为独立 `.o`，最终链接为内核模块

### 总结

- `hw_def.h`：硬件抽象层，定义接口规范
- `sdr.h`：数据结构层，定义共享数据
- `sdr.c`：核心实现，协调各子模块
- `sysfs_intf.c`：用户空间 sysfs 接口
- `sdrctl_intf.c`：测试模式接口
- `Makefile`：编译配置
- `make_all.sh`：构建脚本

## 为什么 side_ch 需要单独处理

### 1. 独立性：可选模块，非核心依赖

主 Makefile 中只包含核心模块：
```bash:3:openwifi-master/driver/Makefile
obj-m += sdr.o openofdm_rx/openofdm_rx.o openofdm_tx/openofdm_tx.o tx_intf/tx_intf.o rx_intf/rx_intf.o xpu/xpu.o
```

`side_ch` 不在其中，因为：
- 不是核心 WiFi 功能的一部分
- 用于高级功能（CSI 捕获、IQ 数据捕获）
- 可以独立编译和使用

### 2. 架构差异：不导出 API 给主驱动

其他模块（如 `tx_intf`、`rx_intf`、`xpu`）：
- 导出驱动 API（如 `tx_intf_driver_api`、`rx_intf_driver_api`）
- 被 `sdr.c` 通过 `extern` 声明使用
- 是主驱动的组成部分

`side_ch` 模块：
- 是独立的平台驱动（`platform_driver`）
- 使用 Netlink 与用户空间通信
- 不导出 API 给 `sdr.c`
- 独立运行，不依赖主驱动

### 3. 构建方式相同，但分离管理

`side_ch/make_driver.sh` 与 `make_all.sh` 中其他模块的构建逻辑相同：
- 设置 Xilinx 环境
- 确定内核路径和交叉编译工具链
- 调用 `make` 编译

单独处理的原因：
- 模块独立性：可选，不影响核心功能
- 编译顺序：可在主驱动之后单独编译
- 维护性：便于独立维护和更新

### 4. 使用场景

`side_ch` 用于：
- 研究用途：CSI 捕获、信道分析
- 调试用途：IQ 数据捕获
- 高级功能：均衡器输出分析

这些功能不是 WiFi 基本功能，因此作为可选模块单独处理。

## 总结

`side_ch` 单独处理的原因：
1. 功能独立性：可选的高级功能模块
2. 架构独立性：不依赖主驱动，也不被主驱动依赖
3. 编译独立性：可以独立编译，不影响核心驱动
4. 维护便利性：便于独立维护和更新

这种设计使核心驱动保持简洁，同时支持高级功能。

## side_ch 模块的选择机制

### 1. 编译阶段的选择

在 `make_all.sh` 中，`side_ch` 的编译是自动执行的，但可以手动控制：

```bash:129:openwifi-master/driver/make_all.sh
cd $OPENWIFI_DIR/driver/side_ch
./make_driver.sh $XILINX_DIR $ARCH_OPTION
```

选择方式：
- 需要 CSI/IQ 功能：保留这两行，正常编译
- 不需要：注释掉这两行，跳过编译

例如：
```bash
# 注释掉这两行来跳过 side_ch 编译
# cd $OPENWIFI_DIR/driver/side_ch
# ./make_driver.sh $XILINX_DIR $ARCH_OPTION
```

### 2. 运行时加载的选择

即使编译了 `side_ch.ko`，也需要手动加载才能使用：

需要 CSI/IQ 功能时：
```bash
insmod side_ch.ko                    # 使用默认参数
insmod side_ch.ko num_eq_init=3     # 自定义均衡器数量（0-8）
insmod side_ch.ko iq_len_init=1000  # 启用 IQ 捕获模式
```

不需要时：
- 不执行 `insmod side_ch.ko`，模块不会加载，不影响核心 WiFi 功能

### 3. 配置参数的选择

加载时可配置模块参数：

```bash:39:openwifi-master/driver/side_ch/side_ch.c
static int num_eq_init = 8; // should be 0~8
static int iq_len_init = 0; //if iq_len>0, iq capture enabled, csi disabled

module_param(num_eq_init, int, 0);
MODULE_PARM_DESC(num_eq_init, "num_eq_init. 0~8. number of equalizer output (52 each) appended to CSI");

module_param(iq_len_init, int, 0);
MODULE_PARM_DESC(iq_len_init, "iq_len_init. if iq_len_init>0, iq capture enabled, csi disabled");
```

- `num_eq_init`：均衡器输出数量（0-8，默认 8）
- `iq_len_init`：IQ 捕获长度（>0 启用 IQ 捕获，禁用 CSI；0 则启用 CSI）

### 4. 使用场景的选择

根据需求选择不同的使用方式：

#### 场景 A：只需要基本 WiFi 功能
```bash
# 编译时跳过 side_ch
# 运行时不加载 side_ch.ko
# 只使用核心驱动模块
```

#### 场景 B：需要 CSI 分析
```bash
# 编译 side_ch.ko
insmod side_ch.ko num_eq_init=8
./side_ch_ctl g                    # 开始捕获 CSI
python3 side_info_display.py       # 显示 CSI 数据
```

#### 场景 C：需要 IQ 数据捕获
```bash
# 编译 side_ch.ko
insmod side_ch.ko iq_len_init=1000  # 启用 IQ 捕获
./side_ch_ctl wh8d8                 # 配置触发条件
./side_ch_ctl g                     # 开始捕获
```

### 5. 设备树支持

设备树中已定义 `side_ch` 设备节点（如 `devicetree.dts`），但模块加载是手动的，不会自动加载。

## 总结

`side_ch` 的选择机制：

| 阶段 | 选择方式 | 说明 |
|------|---------|------|
| **编译** | 在 `make_all.sh` 中注释/取消注释 | 控制是否编译模块 |
| **加载** | 手动执行 `insmod side_ch.ko` | 控制是否加载模块 |
| **配置** | 模块参数 `num_eq_init`、`iq_len_init` | 控制功能模式 |
| **使用** | 用户空间工具 `side_ch_ctl` | 控制捕获行为 |

这种设计的好处：
- 核心驱动保持简洁，不依赖可选功能
- 按需加载，不影响性能
- 灵活配置，适应不同研究需求
- 易于维护，可选模块独立管理

因此，`side_ch` 是真正的可选模块：编译、加载、使用都可以根据需要选择。
